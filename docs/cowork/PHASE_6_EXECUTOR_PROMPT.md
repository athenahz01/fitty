## Mission

Two pieces: (1) **Narrative & Essay Studio** — school-specific, profile-aware essay + activity-list feedback
that is grounded in the target school's stated priorities and patterns from strong similar-admit essays
(RAG), not a generic GPT wrapper; and (2) **Major/Career Compass** — a major ↔ career ↔ outcomes explorer
that ties each major back to the student's real admit odds (ROI later). This is the first phase that calls the
**Anthropic API**, so LLM-safety and "no hallucinated numbers" become first-class. Behind flags, no auto-promote.

## Hard constraints (Cowork will block on these)

1. **No ghostwriting.** The essay tool gives *feedback, diagnostics, and targeted suggestions about the
   student's own text* — it must **never** generate a submittable essay, rewrite the essay wholesale, or
   produce drafted prose the student could paste in. No "write my essay" path. Explicitly do **not** build any
   "AI-detection evasion / humanizer" feature. Audit will read the system prompts + output contract.
2. **Preserve student voice.** Suggestions reference the student's sentences and intent; they don't replace
   the writing with generic polished prose. Feedback is constrained to the student's voice.
3. **Grounded, not boilerplate.** Essay feedback must be grounded in (a) the target school's stated priorities
   (CDS C7 / program supplement prompts) and (b) retrieved patterns from a curated similar-admit essay corpus
   (RAG). The response must carry what it was grounded in (the C7 factors / retrieved exemplar references used)
   so it's traceable — not free-floating GPT output.
4. **No hallucinated numbers — the LLM never emits figures.** Any number shown in Narrative or Compass
   (admit odds, major-level earnings, counts) comes from the deterministic data/model layer and is injected
   into the UI, **not** generated by the model. The Anthropic call produces qualitative text only; numbers are
   passed around it. (LLM prose may be non-deterministic; that's fine for *text* — never for *numbers*.)
5. **Anthropic API safety (first LLM endpoint).** `ANTHROPIC_API_KEY` stays server-side only (never in the
   client bundle); **rate-limit** `/api/narrative` and `/api/compass`; validate input (Zod); **stream** the
   response (house principle). Do **not** log raw essay text; do **not** send PII to the model beyond the
   essay/profile fields the student provides; do **not** persist essays without consent (if stored at all, use
   owner-RLS like Phase 5 — prefer ephemeral/client-held).
6. **Career/major data is sourced.** majors ↔ careers ↔ earnings come from Scorecard major-level earnings +
   an O*NET-style career dataset, each row carrying `source_url` lineage (seed discipline like Phase 0/3). No
   invented salaries or career stats.
7. **Compass admit-odds = the real engine; ROI deferred.** The major→odds link uses the Phase 1 scorer (same
   tier/score as `/api/admit-intelligence`). The ROI/net-cost connection is a clearly-labeled stub ("ROI
   arrives with the Money module") — no placeholder ROI numbers.

## Build

### 1. `lib/narrative/`
- Claude-powered feedback engine: build the prompt from the student's text + the school's CDS C7 priorities +
  RAG hits from a curated, embedded essay-pattern corpus (reuse the Xenova embedding stack). Output contract =
  structured feedback (strengths, gaps vs. the school's priorities, specific line-level suggestions, an
  activity-list/150-char optimizer, a cross-application coherence check) — **not** a drafted essay. Keep the
  prompt-construction pure/testable; isolate the single Anthropic call behind a thin client.
- Curated corpus: committed, provenance-flagged, `source_url` lineage, **no real-student PII**; document it.

### 2. `lib/compass/`
- Deterministic assembler: embed majors ↔ careers ↔ outcomes; recommend major fits; show career paths +
  sourced earnings; connect each major to the student's real admit odds (Phase 1). ROI tie-in stubbed.

### 3. APIs
- `/api/narrative` (streaming, Anthropic, rate-limited) and `/api/compass` (+ status routes), behind
  `ADMIRA_NARRATIVE_ENABLED` and `ADMIRA_COMPASS_ENABLED` (default `"false"`). Zod-validated; server-side keys;
  no raw-essay logging.

### 4. Migration (if needed)
- Career/major + corpus reference tables with `source_url` lineage and public-read RLS (reference data); any
  user-essay storage table (if you add one) gets owner-RLS like Phase 5. `gen_random_uuid()` (no pgcrypto).
  Reversible; applies/reverses cleanly on a scratch DB.

### 5. Frontend
- Essay workspace with inline, specific suggestions (clearly "suggestions," never auto-applied prose); a
  major/career explorer graph. Bold headline first; no hedging copy; ROI shown as "coming with Money."

### 6. Tests (required for sign-off)
- **No-ghostwriting test:** the engine/contract never returns a full essay or whole-essay rewrite; a
  "write my essay" style input is refused/redirected to feedback.
- **Grounding test:** feedback payload references the school's C7 priorities and/or retrieved exemplars it used.
- **No-number-hallucination test:** numbers in Narrative/Compass come from the data/model layer; the Anthropic
  client is mocked and asserted to receive text-only duties (no figures sourced from it).
- **Career-data-lineage test:** every career/earnings row has a `source_url`; none fabricated.
- **Compass-consistency test:** the admit odds shown == `/api/admit-intelligence` for the same profile/school;
  ROI is a labeled stub, not a number.
- **Security tests:** key never in client bundle; `/api/narrative` + `/api/compass` rate-limited; no raw essay
  in logs; (if essays stored) owner-RLS — extend `verify:rls`.
- Playwright e2e for the essay-feedback flow (mocked Anthropic) and the compass explorer (flag on).
